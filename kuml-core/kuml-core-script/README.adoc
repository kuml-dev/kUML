= kuml-core-script
:toc:

image:../../docs/images/kuml-logo-simple.png[kUML,width=48,role=left] *kuml-core-script*

Kotlin Scripting host for `*.kuml.kts` files. Provides `KumlScriptDefinition`
(the `@KotlinScript` annotation with default imports) and `KumlScriptHost`
(the evaluation API used by the CLI, MCP server, and tests).

== How it works

`KumlScriptDefinition` registers `*.kuml.kts` as a Kotlin script type with
the following default imports, so no explicit `import` statements are needed
in script files:

* `io.kuml.core.model.*`
* `io.kuml.core.model.DiagramType.*`
* `io.kuml.core.dsl.*`

`KumlScriptHost` wraps `BasicJvmScriptingHost` and exposes two `eval` overloads
— one for files, one for inline strings.

== Usage

=== Script file (`hello.kuml.kts`)

[source,kotlin]
----
diagram(name = "Hello kUML", type = DiagramType.CLASS) {
    // no imports needed
}
----

=== Evaluating from Kotlin

[source,kotlin]
----
import io.kuml.core.script.KumlScriptHost
import kotlin.script.experimental.api.ScriptDiagnostic
import java.io.File

val result = KumlScriptHost.eval(File("hello.kuml.kts"))

val errors = result.reports.filter { it.severity == ScriptDiagnostic.Severity.ERROR }
check(errors.isEmpty()) { errors.joinToString("\n") }
----

== Classpath note

`KumlScriptHost` uses `dependenciesFromCurrentContext(wholeClasspath = true)`.
This means the JVM classpath at evaluation time is passed to the script compiler.
`kuml-core-model` and `kuml-core-dsl` must therefore be on the calling
application's classpath — which is guaranteed when depending on this module,
since both are declared as `api` dependencies.

== Installation

[source,kotlin]
----
dependencies {
    implementation("io.kuml:kuml-core-script:0.1.0")
    // kuml-core-model and kuml-core-dsl are exposed transitively via api()
}
----
