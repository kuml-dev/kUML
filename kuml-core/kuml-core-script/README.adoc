= kuml-core-script
:toc:

image:../../docs/images/kuml-logo-simple.png[kUML,width=48,role=left] *kuml-core-script*

Kotlin Scripting host for `*.kuml.kts` files. Provides `KumlScriptDefinition`
(the `@KotlinScript` annotation with default imports) and `KumlScriptHost`
(the evaluation API used by the CLI, MCP server, and tests).

== How it works

`KumlScriptDefinition` registers `*.kuml.kts` as a Kotlin script type with
the following default imports, so no explicit `import` statements are needed
in script files:

* `dev.kuml.core.model.*` — `KumlModel`, `KumlDiagram`, `DiagramType`, `ModelLevel`, …
* `dev.kuml.core.model.DiagramType.*` — enum constants
* `dev.kuml.core.dsl.*` — `diagram(…)` entry point
* `dev.kuml.uml.dsl.*` — `umlModel`, `classDiagram`, `classOf`, `interfaceOf`, `enumOf`, `` `package` ``, `association`, …
* `dev.kuml.uml.*` — `UmlClass`, `UmlInterface`, `Visibility`, `AggregationKind`, …
* `dev.kuml.uml.Visibility.*` — visibility enum constants
* `dev.kuml.uml.AggregationKind.*` — aggregation enum constants
* `dev.kuml.c4.dsl.*` *(V1.0+)* — `c4Model`, `person`, `softwareSystem`, `container`, `relationship`, `systemContextDiagram`, …
* `dev.kuml.c4.model.*` *(V1.0+)* — `C4Person`, `C4SoftwareSystem`, `C4Model`, …

`KumlScriptHost` wraps `BasicJvmScriptingHost` and exposes two `eval` overloads
— one for files, one for inline strings.

== Usage

=== Script file (`hello.kuml.kts`)

The project convention (see `CLAUDE.md`) is *named parameters everywhere* and
classifier references as first-class `val`s — not `"namespace::Id"` strings.

[source,kotlin]
----
// No imports needed — everything is in scope via KumlScriptDefinition
classDiagram(name = "Order Domain") {
    val status = enumOf(name = "OrderStatus") {
        literal(name = "DRAFT")
        literal(name = "CONFIRMED")
    }
    val order = classOf(name = "Order") {
        attribute(name = "id", type = "UUID")
        attribute(name = "status", type = status)
        operation(name = "confirm")
    }
    val orderItem = classOf(name = "OrderItem") {
        attribute(name = "quantity", type = "Int")
    }
    association(source = order, target = orderItem) {
        aggregation = AggregationKind.COMPOSITE
        source { multiplicity(spec = "1") }
        target { multiplicity(spec = "1..*") }
    }
}
----

NOTE: Ending the script with `classDiagram { … }` returns a `KumlDiagram`,
which `kuml render` and `DiagramExtractor.extract` can consume directly.
Multi-diagram model files use `umlModel(name = "…") { … }` (returns a
`KumlModel`) or `c4Model(name = "…") { … }` (returns a `C4Model`).

=== Evaluating from Kotlin

[source,kotlin]
----
import dev.kuml.core.script.KumlScriptHost
import kotlin.script.experimental.api.ScriptDiagnostic
import java.io.File

val result = KumlScriptHost.eval(File("order-domain.kuml.kts"))

val errors = result.reports.filter { it.severity == ScriptDiagnostic.Severity.ERROR }
check(errors.isEmpty()) { errors.joinToString("\n") }
----

== Classpath note

`KumlScriptHost` uses `dependenciesFromCurrentContext(wholeClasspath = true)`.
This means the JVM classpath at evaluation time is passed to the script compiler.
`kuml-core-model`, `kuml-core-dsl`, `kuml-metamodel-uml`, and `kuml-metamodel-c4`
must therefore be on the calling application's classpath — which is guaranteed
when depending on this module, since all four are declared as `api`
dependencies transitively.

== Installation

[source,kotlin]
----
dependencies {
    implementation("dev.kuml:kuml-core-script:0.24.0")
    // kuml-core-model, kuml-core-dsl, kuml-metamodel-uml, and kuml-metamodel-c4
    // are exposed transitively via api()
}
----
