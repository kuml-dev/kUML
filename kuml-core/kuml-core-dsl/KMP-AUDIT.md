# kuml-core-dsl KMP Compatibility Audit (V3.2.5)

## Summary
Audited all 54 Kotlin source files in kuml-core-dsl/src/main/kotlin for KMP-blocking java.* usage. Result: the module is exceptionally KMP-clean. There is exactly ONE java.* dependency — `java.util.concurrent.atomic.AtomicLong` in C4Ids.kt — and zero inline java.* usages anywhere else. No java.time, java.math (BigDecimal/BigInteger), java.util.UUID, java.io, java.nio, ServiceLoader, java.util.regex.Pattern, System/Thread/Class.forName, ::class.java, @Jvm* annotations, kotlin.concurrent.thread, ThreadLocal, or synchronized blocks were found. The only external (non-dev.kuml, non-kotlin-stdlib) import in the entire module is that single AtomicLong line. All collection/map usage is Kotlin stdlib (already multiplatform). build.gradle.kts uses the kotlin.jvm plugin with jvmToolchain(21) and depends on api(kuml-core-model), api(kuml-metamodel-uml), api(kuml-metamodel-c4), api(kuml-profile-api), implementation(kotlinx.serialization.json — already KMP-capable), plus kotest test deps. kotlinx-serialization-json is multiplatform, so no replacement needed there. NOTE: the four project() dependencies (kuml-core-model, kuml-metamodel-uml, kuml-metamodel-c4, kuml-profile-api) are SEPARATE modules and were NOT audited here; they must each be KMP-migrated first or in parallel, since this module's api() exposure means it cannot become a multiplatform target until its upstream modules do. This intra-module audit only certifies that kuml-core-dsl's own code introduces a single, trivial blocker.

## Recommendation
V3.2.6 (the actual KMP migration of kuml-core-dsl's own code) is STRAIGHTFORWARD and LOW-RISK. The module's own source has a single trivial blocker (AtomicLong), resolvable by swapping to kotlinx-atomicfu with a near-identical API and no semantic change. There are no date/time, big-number, UUID, IO, reflection, or service-loading dependencies to design expect/actual bridges for. The REAL risk is NOT in this module — it is the dependency chain: because build.gradle.kts re-exports kuml-core-model, kuml-metamodel-uml, kuml-metamodel-c4, and kuml-profile-api via api(), kuml-core-dsl cannot be converted to a kotlin.multiplatform module until those four upstream modules are also multiplatform (a JVM-only api dependency cannot be consumed by non-JVM targets of a KMP module). Therefore the migration order matters far more than the per-line effort. Recommendation: keep V3.2.6 scoped to (a) the atomicfu swap and (b) flipping the build to the multiplatform plugin with a jvm() target only, deferring true commonMain extraction until the upstream model/metamodel/profile modules are audited and migrated. This de-risks the change and keeps the build green at every step.

## V3.2.6 Implementation Plan
1. Step 0 (prerequisite, separate waves): Audit and KMP-migrate the four upstream api() dependencies first — kuml-core-model, kuml-metamodel-uml, kuml-metamodel-c4, kuml-profile-api. kuml-core-dsl re-exports them via api(), so it cannot reach non-JVM targets until they are multiplatform. Run the same java.* audit on each before V3.2.6 proceeds to commonMain.
2. Step 1: Add kotlinx-atomicfu to the version catalog (gradle/libs.versions.toml) — atomicfu library coordinate plus the org.jetbrains.kotlinx.atomicfu Gradle plugin — if not already present from a sibling module.
3. Step 2: In C4Ids.kt, replace `import java.util.concurrent.atomic.AtomicLong` with `import kotlinx.atomicfu.atomic`. Change `private val counter = AtomicLong(0L)` to `private val counter = atomic(0L)`. Update call sites: counter.set(0L) → counter.value = 0L (or counter.getAndSet(0L)); counter.get() → counter.value; counter.getAndIncrement() stays as-is (atomicfu provides getAndIncrement()). Keep all KDoc determinism/thread-safety guarantees — atomicfu preserves identical semantics.
4. Step 3 (interim, recommended for V3.2.6): Keep the build as a JVM-only module but make the atomicfu swap so the code is already platform-agnostic. Run ./gradlew clean check to confirm green (kotest, ktlint).
5. Step 4 (later wave, once upstream modules are KMP): Convert build.gradle.kts from `alias(libs.plugins.kotlin.jvm)` to `alias(libs.plugins.kotlin.multiplatform)`, apply the atomicfu plugin, declare a jvm() target initially, move src/main/kotlin → src/commonMain/kotlin and src/test/kotlin → src/commonTest/kotlin, and switch test deps to kotest's multiplatform/commonTest variants.
6. Step 5: Verify byte-for-byte deterministic ID output is preserved (the determinism guarantee in C4Ids KDoc) by running the existing C4 SVG snapshot/render smoke tests — atomicfu's getAndIncrement is sequentially identical, so output must remain stable.
7. Step 6: After commonMain extraction, follow the Renderer/Layout-Validierungs-Routine if any rendered output is touched, and update Daily Note + V3.2 project file per the standard wave-closure routine.
