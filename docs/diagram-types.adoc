== UML diagrams

V1 supports the five most-used UML diagram types.

=== Class diagram

Structural diagram: classes, interfaces, enums, attributes, operations,
relationships (association, generalization, composition, aggregation,
realization).

[source,kotlin]
----
classDiagram(name = "Order Domain") {
    val customer = classOf(name = "Customer") {
        attribute(name = "id", type = "UUID")
    }
    val order = classOf(name = "Order") {
        attribute(name = "id", type = "UUID")
    }
    association(source = customer, target = order) {
        source { multiplicity(spec = "1") }
        target { multiplicity(spec = "0..*"); role = "orders" }
    }
}
----

Full example: link:../kuml-tests/examples/class-diagram/order-domain.kuml.kts[`order-domain.kuml.kts`]

=== Sequence diagram

Behavioural: lifelines + messages + return arrows.

[source,kotlin]
----
sequenceDiagram(name = "Place Order") {
    val user = participant(name = "User")
    val api = participant(name = "Order API")
    val db = participant(name = "Database")
    user.sendsTo(api, message = "POST /orders")
    api.sendsTo(db, message = "INSERT")
    db.returnsTo(api, message = "id")
    api.returnsTo(user, message = "201 Created")
}
----

Full example: link:../kuml-tests/examples/sequence-diagram/place-order.kuml.kts[`place-order.kuml.kts`]

=== State diagram

Behavioural: state machines with initial / final states and transitions.

[source,kotlin]
----
stateDiagram(name = "Order Lifecycle") {
    val draft = state(name = "Draft")
    val placed = state(name = "Placed")
    val shipped = state(name = "Shipped")
    initial() goesTo draft
    draft goesTo placed on "submit"
    placed goesTo shipped on "ship"
    shipped goesTo final()
}
----

Full example: link:../kuml-tests/examples/state-diagram/order-lifecycle.kuml.kts[`order-lifecycle.kuml.kts`]

=== Use-case diagram

Behavioural: actors + use cases + associations (include/extend).

[source,kotlin]
----
useCaseDiagram(name = "Checkout") {
    val customer = actor(name = "Customer")
    val browse = useCase(name = "Browse Catalog")
    val checkout = useCase(name = "Place Order")
    val pay = useCase(name = "Pay")
    customer associates browse
    customer associates checkout
    checkout includes pay
}
----

Full example: link:../kuml-tests/examples/use-case-diagram/checkout.kuml.kts[`checkout.kuml.kts`]

=== Component diagram

Structural: components + provided/required interfaces + dependencies.

[source,kotlin]
----
componentDiagram(name = "Architecture") {
    val web = component(name = "Web UI")
    val api = component(name = "API Gateway")
    val orders = component(name = "Orders Service")
    web dependsOn api
    api dependsOn orders
}
----

Full example: link:../kuml-tests/examples/component-diagram/architecture.kuml.kts[`architecture.kuml.kts`]

=== Object diagram (V1.1)

Structural snapshot: instances of classifiers with their slot values, connected
by links. Useful for illustrating a specific configuration the class diagram
permits, or for capturing test fixtures pictorially.

[source,kotlin]
----
// Build classifiers directly (or capture them from a surrounding class diagram).
val customer = UmlClass(
    id = "Customer", name = "Customer",
    attributes = listOf(
        UmlProperty(id = "Customer::name", name = "name", type = UmlTypeRef("String")),
    ),
)

objectDiagram(name = "Order #42 snapshot") {
    val alice = instanceOf(classifier = customer, name = "alice") {
        slot(feature = "name", value = literal("\"Alice\""))
    }
    val bob = instanceOf(classifier = customer, name = "bob") {
        slot(feature = "name", value = literal("\"Bob\""))
    }
    link(from = alice, to = bob, targetRole = "friend")
}
----

Rendered notation:

* Instance header `name : Classifier` — underlined per UML 2.x convention.
* Anonymous instances render as `: Classifier`.
* Slot compartment lists `feature = value` per slot.
* Slot values use the helpers `literal(text)`, `ref(otherInstance)`, or
  `nullValue` for explicit absence.
* Links are plain lines (no arrowhead) with optional `sourceRole` / `targetRole`
  labels at the quarter / three-quarter points.

Full example: link:../kuml-tests/examples/object-diagram/order-snapshot.kuml.kts[`order-snapshot.kuml.kts`]

=== Package diagram (V1.1)

Groups model elements into named folders with `«import»` / `«merge»` /
`«access»` dependencies between them.

[source,kotlin]
----
packageDiagram(name = "Three-Layer Architecture") {
    val web = packageOf(name = "web")
    val service = packageOf(name = "service")
    val repository = packageOf(name = "repository")
    packageImport(client = web, supplier = service)
    packageImport(client = service, supplier = repository)
}
----

Full example: link:../kuml-tests/examples/package/three-layers.kuml.kts[`three-layers.kuml.kts`]

=== Composite-structure diagram (V1.1)

Shows the internal wiring of a classifier — ports, parts and the connectors
between them.

[source,kotlin]
----
compositeStructureDiagram(name = "OrderService — internal structure") {
    val placeOrder = interfaceOf(name = "IPlaceOrder")
    val orderSvc = component(name = "OrderService") {
        port(name = "api"); provides(iface = placeOrder)
    }
    // … additional components + connect(end1, port1, end2, port2)
}
----

Full example: link:../kuml-tests/examples/composite-structure/order-service.kuml.kts[`order-service.kuml.kts`]

=== Deployment diagram (V1.1)

Runtime topology: nodes (hardware, devices, execution environments) hosting
artifacts, linked by communication paths.

[source,kotlin]
----
deploymentDiagram(name = "AWS — production") {
    val vpc = executionEnvironment(name = "VPC eu-central-1") {
        node(name = "EKS Cluster") {
            artifact(name = "kuml-web.jar")
            artifact(name = "kuml-mcp.jar")
        }
    }
    val db = device(name = "RDS PostgreSQL")
    communicationPath(end1 = vpc, end2 = db)
}
----

Notation: nodes render as 3D cubes with `«executionEnvironment»` /
`«device»` stereotypes where set, artifacts as rectangles with a
dog-eared corner.

Full example: link:../kuml-tests/examples/deployment/aws-eks.kuml.kts[`aws-eks.kuml.kts`]

=== Profile diagram (V1.1)

Declares stereotypes with tagged-value definitions and `«extension»`
dependencies to metaclass placeholders.

[source,kotlin]
----
profileDiagram(name = "Java EE profile") {
    val entity = stereotype(name = "Entity", metaclasses = listOf("Class")) {
        tag(name = "tableName", type = "String")
    }
    val cls = metaclass(name = "Class")
    extension(stereotype = entity, metaclass = cls)
}
----

Full example: link:../kuml-tests/examples/profile/javaee.kuml.kts[`javaee.kuml.kts`]

=== Activity diagram (V1.1)

Control-flow + object-flow graph. Initial / final, actions, decisions,
forks/joins, and guarded edges.

[source,kotlin]
----
activityDiagram(name = "Order checkout") {
    val start = initialNode()
    val validate = action(name = "Validate cart")
    val ok = decision(name = "valid?")
    val charge = action(name = "Charge card")
    val end = finalNode()
    edge(from = start, to = validate)
    edge(from = validate, to = ok)
    edge(from = ok, to = charge, guard = "valid")
    edge(from = charge, to = end)
}
----

Edges use `guard = "…"` for labelled branches and `objectFlow = true` for
dashed object flow lines.

Full example: link:../kuml-tests/examples/activity/order-checkout.kuml.kts[`order-checkout.kuml.kts`]

=== Communication diagram (V1.1)

The same scenario as a sequence diagram but laid out spatially. Each
`message(from, to, label)` call gets an auto-assigned sequence number.

[source,kotlin]
----
communicationDiagram(name = "Place Order — communication view") {
    val ui = role(classifierName = "UI", roleName = "ui")
    val api = role(classifierName = "OrderApi", roleName = "api")
    val db = role(classifierName = "Database", roleName = "db")
    message(from = ui, to = api, label = "POST /orders")
    message(from = api, to = db, label = "INSERT")
    message(from = api, to = ui, label = "201 Created")
}
----

Full example: link:../kuml-tests/examples/communication/place-order.kuml.kts[`place-order.kuml.kts`]

=== Timing diagram (V1.1)

State of one or more lifelines over discrete time. Each `tick(t, state)`
records a state transition; the renderer joins ticks with a step-line.

[source,kotlin]
----
timingDiagram(name = "TCP three-way handshake") {
    lifeline(name = "client", states = listOf("CLOSED", "SYN_SENT", "ESTABLISHED")) {
        tick(t = 0, state = "CLOSED")
        tick(t = 1, state = "SYN_SENT")
        tick(t = 4, state = "ESTABLISHED")
    }
    // … server lifeline
}
----

Full example: link:../kuml-tests/examples/timing/tcp-handshake.kuml.kts[`tcp-handshake.kuml.kts`]

=== Interaction-overview diagram (V1.1)

Activity-style control flow over `ref InteractionName` frames that point
at full sequence / communication diagrams elsewhere.

[source,kotlin]
----
interactionOverviewDiagram(name = "Order overview") {
    val init = initial()
    val login = interactionRef(name = "Login")
    val checkout = interactionRef(name = "Checkout")
    val end = final()
    edge(from = init, to = login)
    edge(from = login, to = checkout)
    edge(from = checkout, to = end)
}
----

Full example: link:../kuml-tests/examples/interaction-overview/order-overview.kuml.kts[`order-overview.kuml.kts`]

== C4 diagrams

The C4 metamodel ships in full — five C4 levels plus the dynamic-flow view.

=== System Context (Level 1)

[source,kotlin]
----
c4Model(name = "Banking") {
    val customer = person(name = "Customer")
    val system = softwareSystem(name = "Online Banking")
    relationship(source = customer, target = system) { technology = "HTTPS" }
    systemContextDiagram(name = "Context") {
        include(customer, system)
    }
}
----

Full example: link:../kuml-tests/examples/c4/system-context/internet-banking.kuml.kts[`internet-banking.kuml.kts`]

=== Container (Level 2)

Shows the containers (apps, databases) inside a software system.

[source,kotlin]
----
c4Model(name = "Banking") {
    val customer = person(name = "Customer")
    val system = softwareSystem(name = "Online Banking") {
        val web = container(name = "Web Application", technology = "Kotlin/Ktor")
        val db = container(name = "Database", technology = "PostgreSQL")
        web.uses(db, technology = "JDBC")
    }
    containerDiagram(name = "Banking — Containers") {
        include(customer, system)
    }
}
----

Full example: link:../kuml-tests/examples/c4/core/internet-banking.kuml.kts[`internet-banking.kuml.kts`]

=== Component (Level 3)

Drills into a single container.

[source,kotlin]
----
c4Model(name = "API") {
    val api = softwareSystem(name = "API") {
        val auth = container(name = "Auth Service") {
            component(name = "TokenService", technology = "Kotlin")
            component(name = "UserRepo", technology = "Exposed")
        }
    }
    componentDiagram(name = "Auth — Components") { /* … */ }
}
----

Full example: link:../kuml-tests/examples/c4/component/api-components.kuml.kts[`api-components.kuml.kts`]

=== System Landscape

Broad overview across multiple software systems.

[source,kotlin]
----
c4Model(name = "Enterprise") {
    val crm = softwareSystem(name = "CRM")
    val erp = softwareSystem(name = "ERP")
    val mail = softwareSystem(name = "Email") { external = true }
    relationship(source = crm, target = erp)
    relationship(source = crm, target = mail)
    systemLandscapeDiagram(name = "Landscape") { include(crm, erp, mail) }
}
----

=== Deployment

Maps containers to infrastructure (nodes, regions).

[source,kotlin]
----
c4Model(name = "Banking") {
    deploymentDiagram(name = "Production") {
        deploymentNode(name = "AWS eu-central-1") {
            deploymentNode(name = "EKS Cluster") {
                containerInstance(name = "web-app")
            }
        }
    }
}
----

Full example: link:../kuml-tests/examples/c4/deployment/kubernetes-deployment.kuml.kts[`kubernetes-deployment.kuml.kts`]

=== Dynamic

Sequence-like view of interactions between elements.

[source,kotlin]
----
c4Model(name = "Order Flow") {
    val user = person(name = "User")
    val web = softwareSystem(name = "Web")
    val db = softwareSystem(name = "Database")
    dynamicDiagram(name = "Checkout") {
        interaction(description = "POST /order", from = user, to = web)
        interaction(description = "INSERT",       from = web,  to = db)
        response(description = "id",              from = db,   to = web)
        response(description = "201",             from = web,  to = user)
    }
}
----

Full example: link:../kuml-tests/examples/c4/dynamic/order-processing.kuml.kts[`order-processing.kuml.kts`]
