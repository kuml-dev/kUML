= kUML Examples
:toc:
:toc-placement: preamble
:icons: font

This directory contains runnable kUML example scripts demonstrating the full feature surface — rendering, simulation, and model-to-model code generation.

== Directory Overview

[cols="1,2,2,3", options="header"]
|===
| Directory | Domain | Diagram types | Special notes

| `autosar/`
| AUTOSAR Classic Platform (R22-11)
| Class (with AUTOSAR profile stereotypes)
| `«SoftwareComponent»`, `«ComInterface»`, `«Runnable»`, `«BehaviorSpec»`; AUTOSAR port direction D17 convention

| `c4/`
| Generic system architecture
| C4 Container, C4 Landscape
| Demonstrates kUML's C4 first-class support

| `javaee/`
| JavaEE / Jakarta EE applications
| Class (JavaEE profile)
| Persistence context, user domain with `«Entity»`, `«SessionBean»` stereotypes

| `jpa/`
| JPA entity generation
| Class (UML → JPA M2M)
| `blog-domain.kuml.kts` is the canonical `kuml transform` target; generates Kotlin JPA entity files via `--transformer uml-to-jpa`

| `openapi/`
| REST API specification
| Class (OpenAPI profile)
| Feature operations, user resource with `«Resource»`, `«Operation»` stereotypes

| `pepela/`
| Pepela Smart Home (thermostat)
| SysML 2 STM + SysML 2 ACT
| Thermostat state machine and boot-flow activity; *both* are runnable via `kuml simulate`

| `soaml/`
| Service-oriented architecture
| Class (SoaML profile)
| Order processing service composition with `«ServiceInterface»`, `«Service»` stereotypes

| `spring/`
| Spring Framework applications
| Class (Spring profile)
| Payment service and scheduled tasks with `«Component»`, `«Service»`, `«Scheduled»` stereotypes

| `sysml2/`
| SysML 2 across all supported diagram types
| BDD, UC, PAR, ACT, STM, SEQ, REQ
| Covers all 7 simulatable/renderable SysML 2 diagram types; ACT and STM are runnable via `kuml simulate`

| `rest/`
| REST API code generation
| Class (UML → OpenAPI)
| `blog-api.kuml.kts` generates an OpenAPI 3.0 YAML spec via `--transformer uml-to-rest`

| `k8s/`
| Kubernetes deployment generation
| Component (UML → K8s)
| `order-system-k8s.kuml.kts` generates Deployment + Service manifests via `--transformer uml-to-k8s`

| `docker/`
| Dockerfile generation
| Component (UML → Docker)
| `order-system-docker.kuml.kts` generates a Dockerfile per component via `--transformer uml-to-docker`

| `c4-to-uml/`
| C4 → UML round-trip
| C4 → UML class diagram script
| `banking-to-uml.kuml.kts` converts a C4 model to a UML class diagram script via `--transformer c4-to-uml`

| `keysight/`
| Keysight Car2x / V2X automotive scenario
| SysML 2 STM + SysML 2 BDD
| V2X intersection scenario (Idle → Approaching → Negotiating → Crossing → Departed); three event files; runnable via `kuml simulate`

| `pzb/`
| PZB Datenbank-Schema (Exposed-Tabellen)
| Class
| Reales 19-Tabellen-Exposed-Schema des PZB-Projekts. Dient als anspruchsvolles Renderer-Test-Fixture (abstrakte Basisklasse, viele Generalisations-Kanten zu einem gemeinsamen Parent, parallele FK-Kanten zwischen identischen Klassen-Paaren, mehrere Self-Loops über `optReference parent → self`, `1`- und `0..1`-Multiplicities)

| `desktop/`
| kuml-desktop itself (self-referential dogfooding)
| Use Case
| `kuml-desktop-use-cases.kuml.kts` models the actors and implemented use cases of the `kuml-desktop` application (editing/rendering, AI-assisted modeling, knowledge-workspace browsing, plugin discovery); actor generalization plus `«include»`/`«extend»` relationships between use cases
|===

== Runnable Categories

=== `kuml render` — all examples

Every `.kuml.kts` file in this tree can be rendered to SVG or PNG:

[source,bash]
----
kuml render autosar/autosar-engine-control.kuml.kts --format svg
kuml render sysml2/hybrid-vehicle.kuml.kts --format svg
kuml render pepela/pepela-thermostat-stm.kuml.kts --format png
----

=== `kuml simulate` — behavioural execution

Simulation is supported for STM (state-machine semantics) and ACT (token-flow activity semantics) in both UML and SysML 2.

[cols="2,2,2", options="header"]
|===
| Script | Events file | What it exercises

| `pepela/pepela-thermostat-stm.kuml.kts`
| `pepela/pepela-thermostat-stm-events.json`
| SysML 2 STM — thermostat state machine (Off → Heating → Idle → Cooling)

| `pepela/pepela-thermostat-flow.kuml.kts`
| `pepela/pepela-thermostat-flow-events.json`
| SysML 2 ACT — thermostat boot-flow with token propagation

| `sysml2/traffic-light-stm.kuml.kts`
| _(inline initial event)_
| SysML 2 STM — classic traffic light cycle

| `sysml2/order-processing-act.kuml.kts`
| `sysml2/order-processing-act-events.json`
| SysML 2 ACT — order processing activity with expected trace verification

| `keysight/car2x-scenario-stm.kuml.kts`
| `keysight/car2x-events-happy.json`
| SysML 2 STM — V2X intersection: Idle → Approaching → Negotiating → Crossing → Departed (happy path)

| `keysight/car2x-scenario-stm.kuml.kts`
| `keysight/car2x-events-conflict.json`
| SysML 2 STM — V2X conflict path (yields back to Idle)

| `keysight/car2x-scenario-stm.kuml.kts`
| `keysight/car2x-events-late-detection.json`
| SysML 2 STM — late detection (TTC guard fails, stays in Idle)
|===

[source,bash]
----
kuml simulate pepela/pepela-thermostat-stm.kuml.kts \
    --events pepela/pepela-thermostat-stm-events.json

kuml simulate sysml2/order-processing-act.kuml.kts \
    --events sysml2/order-processing-act-events.json
----

=== `kuml transform` — M2M code generation

[cols="2,2,2", options="header"]
|===
| Script | Transformer | Output

| `jpa/blog-domain.kuml.kts`
| `uml-to-jpa`
| Kotlin JPA `@Entity` files (one `.kt` per class)

| `rest/blog-api.kuml.kts`
| `uml-to-rest`
| OpenAPI 3.0 YAML (`openapi.yaml`) with CRUD endpoints per class

| `k8s/order-system-k8s.kuml.kts`
| `uml-to-k8s`
| Kubernetes `deployment.yaml` per component (Deployment + Service)

| `docker/order-system-docker.kuml.kts`
| `uml-to-docker`
| `Dockerfile` per component (eclipse-temurin:21-jre-alpine base)

| `c4-to-uml/banking-to-uml.kuml.kts`
| `c4-to-uml`
| `.kuml.kts` script reproducing the C4 model as a UML class diagram
|===

[source,bash]
----
kuml transform jpa/blog-domain.kuml.kts \
    --transformer uml-to-jpa --output generated/jpa/

kuml transform rest/blog-api.kuml.kts \
    --transformer uml-to-rest --output generated/rest/

kuml transform k8s/order-system-k8s.kuml.kts \
    --transformer uml-to-k8s --output generated/k8s/

kuml transform docker/order-system-docker.kuml.kts \
    --transformer uml-to-docker --output generated/docker/

# List all registered transformers (shows uml-to-jpa, uml-to-rest, uml-to-k8s, uml-to-docker, c4-to-uml):
kuml transform --list-transformers
----

== Quick Start

Five one-liners to see kUML in action immediately:

[source,bash]
----
# 1. Render a SysML 2 hybrid-vehicle block definition diagram to SVG
kuml render sysml2/hybrid-vehicle.kuml.kts --format svg

# 2. Simulate a thermostat state machine end-to-end
kuml simulate pepela/pepela-thermostat-stm.kuml.kts \
    --events pepela/pepela-thermostat-stm-events.json

# 3. Generate Kotlin JPA entities from a UML class diagram
kuml transform jpa/blog-domain.kuml.kts \
    --transformer uml-to-jpa --output /tmp/jpa-gen/

# 4. Simulate a real-world automotive V2X intersection scenario
kuml simulate keysight/car2x-scenario-stm.kuml.kts \
    --events keysight/car2x-events-happy.json

# 5. Generate OpenAPI 3.0 YAML from a UML class diagram
kuml transform rest/blog-api.kuml.kts \
    --transformer uml-to-rest --output /tmp/rest-gen/
----
